¿Por que aunque tenga la IP de un dominio no puedo ingresar a la página a través de la IP?

May 06, 2026
Dns

DeepSeek:

Esa es una pregunta excelente y muy común. El hecho de que tengas la dirección IP de un dominio pero no puedas acceder al sitio web a través de ella se debe a varias razones técnicas, siendo la más importante el Hosting Virtual (Virtual Hosting).

Vamos a desglosarlo con una analogía y la explicación técnica.

1. La razón principal: Hosting Virtual Compartido

Imagina un gran edificio de oficinas (el servidor) con una sola dirección postal (la IP pública). Dentro de ese edificio, trabajan cientos de empresas diferentes (los sitios web). Si tú envías una carta solo con la dirección del edificio, sin poner el nombre de la empresa específica a la que va dirigida, el cartero no sabrá a qué oficina entregarla.

  • La analogía técnica:
    • Edificio: El servidor físico con una IP (ej. 192.0.2.10).
    • Empresas: misitio.com, tublog.org, tienda.net.
    • Nombre en la carta: La cabecera Host en tu petición web.

Cuando escribes http://misitio.com en tu navegador, ocurre esto:

  1. Tu navegador resuelve el dominio a la IP 192.0.2.10.
  2. Se conecta a esa IP.
  3. Aquí está la clave: En la petición HTTP, tu navegador envía una cabecera llamada Host: misitio.com.
  4. El servidor web (Apache, Nginx, etc.) lee esa cabecera y dice: “Ah, quieres ver el contenido de la empresa misitio.com”, y te entrega los archivos de ese sitio.

Cuando pones la IP directamente (http://192.0.2.10), la petición se envía sin la cabecera Host válida o con la IP como cabecera. El servidor web no sabe qué sitio entregarte de los cientos que aloja.

  • ¿Qué hace entonces el servidor?
    • Muestra un sitio por defecto: Puede mostrar el primer sitio configurado en su lista, un panel de control del hosting o una página genérica de “¡Funciona!”.
    • Devuelve un error: Un error 404 Not Found o 400 Bad Request porque no tiene un sitio configurado para responder a esa solicitud “sin nombre”.

2. Otras razones importantes

Aunque el hosting virtual es la causa principal, existen otros motivos técnicos y de seguridad:

  • Proxy Inverso y CDNs (Redes de Entrega de Contenido): Muchos sitios grandes no exponen la IP real de su servidor de origen. Usan servicios como Cloudflare, Akamai o un balanceador de carga. La IP que ves para el dominio (104.26.9.239 para Cloudflare, por ejemplo) es la del proxy, no la del servidor final. Este proxy exige recibir la cabecera Host correcta para saber a cuál de sus millones de clientes enrutar la petición. Sin ella, se pierde.

  • Certificados SSL/TLS: Si intentas acceder por IP a un sitio que usa https://, el certificado de seguridad está emitido para el nombre de dominio (ej. *.misitio.com), no para la dirección IP. El navegador te mostrará una aterradora advertencia de seguridad que tendrás que saltarte manualmente, y aun así, el hosting virtual probablemente siga impidiendo la carga correcta.

  • Firewalls de Aplicaciones Web (WAF): Muchos firewalls modernos bloquean directamente cualquier petición que no contenga una cabecera Host reconocida en su lista blanca. Es una medida de seguridad básica contra escaneos automáticos y bots maliciosos, que a menudo buscan vulnerabilidades usando solo IPs.

Conclusión y cómo podrías forzarlo (solo para pruebas)

El sistema de nombres de dominio no es solo una agenda telefónica bonita; el nombre en sí mismo es una parte indispensable de la comunicación para la gran mayoría de las webs modernas.

Si por alguna razón técnica necesitas acceder al sitio usando solo la IP (por ejemplo, para probar algo antes de apuntar un dominio), puedes “engañar” a tu ordenador para que envíe la cabecera Host correcta:

  1. Editando el archivo hosts de tu sistema.
    • En Windows, el archivo está en C:\Windows\System32\drivers\etc\hosts.
    • En macOS y Linux, en /etc/hosts.
  2. Añades una línea como: 192.0.2.10 misitio.com
  3. Guardas el archivo. Ahora, cuando escribas misitio.com en tu navegador, tu ordenador usará esa IP localmente, pero la petición HTTP seguirá llevando la cabecera Host: misitio.com, por lo que el servidor te entenderá perfectamente.

Esto simula un DNS local solo para tu máquina y te permite probar el acceso “por IP” de forma correcta.
