Estos pasos no funcionan tenes que compilar el kernel

Y si yo instalo eso en una distro como Trisquel no me alcanza con hacer eso no ? Tengo que activarlo en el kernel y volver a compilar el kernel no ?

No, no necesitás recompilar el kernel. La mayoría de los drivers de WiFi en Linux son módulos del kernel, no están compilados dentro del kernel estático.

Lo que tendrías que hacer en Trisquel es más simple:

1. Instalar el firmware Simplemente copiar el archivo rtl8723befw.bin a /lib/firmware/rtlwifi/. Podés sacarlo de otro sistema que lo tenga, o del paquete firmware-realtek de Debian.

2. Cargar el módulo

modprobe rtl8723be

Y listo. El módulo rtl8723be o rtl8xxxu ya está compilado en el kernel de Trisquel como módulo, simplemente no se activa porque no encuentra el firmware al arrancar.

Sin embargo hay un detalle importante: Trisquel usa linux-libre, que no solo excluye el firmware sino que también parchea el kernel para eliminar el código que solicita ese firmware. Entonces puede que el módulo directamente no exista o esté parcheado para no funcionar aunque pongas el firmware.

En ese caso sí tendrías dos opciones:

  • Instalar un kernel estándar de Ubuntu/Debian junto a linux-libre
  • O usar directamente una distro que no sea aprobada por la FSF pero sea libre en todo lo demás, como Debian con non-free activado

En Trisquel, una vez que copiaste el firmware a /lib/firmware/rtlwifi/, harías:

sudo modprobe rtl8723be

Eso le dice al kernel que cargue el módulo. Si el módulo encuentra el firmware en la ruta correcta, debería levantar la interfaz WiFi.

Para verificar si funcionó:

# Ver si el módulo se cargó
lsmod | grep rtl8723

# Ver si apareció la interfaz de red

Si el módulo ya estaba cargado y querés recargarlo después de poner el firmware:

sudo modprobe -r rtl8723be  # descarga el módulo
sudo modprobe rtl8723be     # lo vuelve a cargar

Pero como te decía, el problema en Trisquel es anterior a esto: si linux-libre parcheó el módulo para que no solicite firmware propietario, el modprobe puede fallar o cargar el módulo pero sin WiFi funcional. Ahí el error que te tire va a decir exactamente qué pasó.

Después en Trisquel lo copiás a:

sudo cp rtl8723befw.bin /lib/firmware/rtlwifi/

Pero acordate de lo que te decía antes: si Trisquel usa linux-libre parcheado, puede que el módulo directamente no esté disponible y el firmware solo no alcance. Probalo y fijate qué pasa, si no funciona haceme saber el error y vemos.

Para que el módulo se cargue automáticamente al iniciar en Trisquel, agregalo a /etc/modules-load.d/.

Por ejemplo:

Eso hace que systemd-modules-load cargue rtl8723be en cada arranque.

También podés usar el método viejo editando /etc/modules:

echo rtl8723be | sudo sudo tee -a /etc/modules

En distros basadas en linux-libre como Trisquel, muchas veces el módulo existe pero el soporte de firmware propietario está deshabilitado/parcheado. Entonces aunque copies el .bin, el driver puede ignorarlo o negarse a inicializar el hardware. dmesg te va a mostrar exactamente eso si pasa.

explicacion

Exacto, esa es la distinción clave que mucha gente confunde, y yo mismo la expliqué mal al principio de la conversación.

El driver es código que corre en el kernel de tu CPU. En el caso del RTL8723BE, el driver rtl8xxxu es libre y está en el kernel.

El firmware es un programa binario que se carga dentro del chip mismo, en su procesador interno. Ese es el blob propietario. Sin él el chip no sabe hacer nada, no importa qué tan libre sea el driver.

La analogía sería: el driver es el traductor que habla con el chip, pero el firmware es el cerebro del chip. Sin cerebro no hay nada que traducir.

Por eso las distros aprobadas por la FSF funcionan bien con Atheros AR9271: ese chip no necesita que le cargues un firmware externo porque ya viene con todo lo necesario grabado de fábrica en memoria ROM, o el firmware que necesita es libre. En cambio Realtek, Broadcom y la mayoría de los fabricantes modernos diseñaron sus chips para requerir que el sistema operativo les cargue el firmware en cada arranque, y ese firmware es propietario.

Es una distinción técnica importante y vale la pena tenerla clara.

El problema con RTL8723BE y Linux libre

Lamentablemente, es imposible usar la RTL8723BE con una distribución 100% libre (como Trisquel, Parabola, Guix, etc.) sin comprometer los principios de software libre. Te explico por qué:

¿Por qué no funciona?

El chip Realtek RTL8723BE requiere firmware propietario (un blob binario) para funcionar. Este firmware:

  • Es código compilado y cerrado que Realtek no ha liberado con código fuente.
  • Se carga en el chip cada vez que el sistema arranca.
  • Las distribuciones aprobadas por la FSF lo excluyen deliberadamente del kernel y los repositorios.

Sin ese blob, el kernel libre simplemente no puede inicializar el hardware. No es un problema del driver en sí (el driver rtl8723be existe en el kernel), sino del firmware que el chip exige.

¿Por qué la FSF no puede incluirlo?

La FSF considera que los blobs binarios violan las 4 libertades del software libre: no podés estudiarlos, modificarlos ni redistribuirlos libremente. Por eso linux-libre (el kernel que usan esas distros) parchea activamente el kernel oficial para eliminar toda llamada a firmware no libre.

¿Qué podés hacer?

✅ Opción 1 — Reemplazar el adaptador WiFi (recomendada)

Comprar un adaptador WiFi compatible con software 100% libre. Las mejores opciones son:

Chipset Tipo Notas
Atheros AR9271 USB El más recomendado por la FSF/FSF-endorsed distros
Atheros AR7010 USB Funciona con ath9k_htc (libre)
Libertas 88W8388 USB Soportado por el driver libre libertas

El AR9271 es prácticamente el estándar de facto para usuarios de distros libres. Buscá adaptadores USB con ese chipset (marcas como ThinkPenguin, o genéricos en MercadoLibre/AliExpress verificando el chipset).

✅ Opción 2 — Ethernet por cable

Mientras conseguís el adaptador, usá conexión por cable. Prácticamente todos los chips Ethernet integrados funcionan con drivers libres.

✅ Opción 3 — USB tethering desde el celular

Compartí la conexión del celular por USB, que generalmente funciona sin firmware adicional.

❌ Lo que NO funciona

No existe forma de hacer funcionar la RTL8723BE en una distro aprobada por la FSF sin:

  • Añadir el firmware propietario (lo que rompería la pureza de la distro)
  • O que Realtek libere el firmware (algo que no ha ocurrido y es poco probable)

Conclusión: La solución real es cambiar el hardware. El adaptador Atheros AR9271 cuesta muy poco y es la solución estándar de la comunidad de software libre para exactamente este problema.

¿Por qué no se puede simplemente escribir un driver libre?

Es una pregunta muy válida y la respuesta tiene varias capas:

El driver ya existe y es libre… el problema es el firmware

Acá está la confusión clave que mucha gente tiene:

Driver (software en el kernel)  ←→  Firmware (código que corre DENTRO del chip)
       ✅ Ya es libre                      ❌ Este es el problema

El driver rtl8723be que vive en el kernel Linux ya es código abierto. Yo podría estudiarlo ahora mismo. El problema no es el driver, sino el firmware que se carga en el microcontrolador interno del chip.

¿Qué es ese firmware exactamente?

El chip RTL8723BE tiene adentro su propio procesador. Cuando arranca:

  1. El driver lo detecta
  2. Le carga un programa (el blob) a ese procesador interno
  3. Ese programa le enseña al chip cómo hablar WiFi, Bluetooth, manejar la radio, etc.

Sin ese programa, el chip es básicamente un ladrillo. No sabe hacer nada.

¿Y no se puede reescribir ese firmware desde cero?

En teoría sí. En la práctica es enormemente difícil:

  • Realtek no publica la arquitectura del procesador interno ni el conjunto de instrucciones
  • No hay documentación del hardware (registros, interrupciones, protocolo interno)
  • Requeriría años de trabajo de ingenieros especializados en hardware
  • El resultado podría ser inferior o inestable sin documentación oficial

Esto no es como reescribir una aplicación. Es como intentar construir un sistema operativo para una CPU cuya arquitectura es secreta.

¿Y la ingeniería inversa al blob?

Tenés razón en que se intentó. El proyecto OpenFWWF y comunidades como la de ath9k hicieron esto con otros chips. Con Realtek el resultado fue:

  • El blob está ofuscado intencionalmente
  • El procesador interno usa una arquitectura poco documentada
  • Lo que se encontró fueron tablas de calibración de radio y código de control de señal muy específico del hardware
  • No es simplemente “algoritmos WiFi” que se puedan reimplementar limpiamente

Básicamente, sin los esquemáticos del chip es casi imposible hacer ingeniería inversa útil.

¿Hay algún chip con firmware libre?

Sí, y por eso el AR9271 es tan valorado:

Chip Firmware
Realtek RTL8723BE Propietario, cerrado
Atheros AR9271 Libre, incluido en linux-libre
Broadcom BCM43xx Propietario, peor que Realtek

Atheros/Qualcomm en su momento decidió liberar la especificación y el firmware. Realtek nunca lo hizo. Es una decisión empresarial, no técnica.

Conclusión

No es que nadie haya intentado. Es que:

  1. El driver ya es libre, ese no es el problema
  2. Reescribir el firmware requiere documentación que Realtek no da
  3. La ingeniería inversa se intentó y el chip está demasiado cerrado para obtener algo útil
  4. La solución real sigue siendo cambiar el hardware por uno con firmware libre

La lucha por hardware libre es exactamente esto: no alcanza con que el software sea libre si el hardware exige código propietario para funcionar. Es uno de los desafíos más grandes que enfrenta el movimiento del software libre hoy. Tenés razón en cuestionarme, y debo ser honesto: me expresé con más certeza de la que tenía.

Lo que sí es cierto

Realtek ha publicado datasheets de algunos de sus chips, y hay información parcial disponible. Además el proyecto rtl8xxxu (driver alternativo en el kernel) fue desarrollado en parte mediante ingeniería inversa exitosa. Eso contradice lo que dije.

Lo que realmente frena un firmware libre para el RTL8723BE

El problema más preciso no es ausencia total de documentación, sino:

  • El firmware controla la capa PHY (la radio en sí), que involucra calibración muy específica de cada revisión del chip. Sin los parámetros exactos de Realtek, una reimplementación podría funcionar mal o violar regulaciones de radio.
  • Las especificaciones completas del procesador interno del chip (no el chip en general) siguen siendo opacas.
  • Nadie ha puesto el trabajo sostenido necesario. No es imposible en principio, es que requiere mucho esfuerzo con poco incentivo cuando el AR9271 ya existe como solución.

Lo que debí haber dicho

En lugar de “no hay documentación”, lo correcto era: “la documentación disponible es insuficiente para reimplementar el firmware de forma confiable, y el esfuerzo no se ha priorizado porque hay alternativas libres”.

Gracias por el control, es exactamente el tipo de crítica que ayuda a no propagar información incorrecta.

Firmware vs Driver

  • Driver (controlador): el software que controla el hardware. En Linux hay drivers abiertos y cerrados.
  • Firmware: software que corre dentro del chip WiFi mismo. Este casi siempre es binario/propietario porque lo escribe el fabricante del chip.

Puede el firmware espiarte?

Teóricamente sí, pero en la práctica:

  1. El firmware tiene acceso a:

    • Todo el tráfico WiFi que pasa por tu tarjeta
    • Datos que envías/recibes

  2. Lo que NO puede hacer:

    • Acceder a tu disco duro
    • Escapar más allá de su scope de red

  3. Mitigaciones:

    • Usar HTTPS siempre (el tráfico está cifrado punto a punto)
    • VPN (el firmware solo ve tráfico cifrado hacia el servidor VPN)

Alternativas más libres

  • Sin blobs propietarios: solo con hardware que tenga drivers Y firmware completamente libres (ej: algunos routers con OpenWRT, algunas tarjetas WiFi con drivers libres)
  • Airgapping: separar redes sensibles
  • WiFi externo USB: podes usar un adaptador WiFi con firmware más transparente

Resumen

Usar blobs propietarios de WiFi es un riesgo de privacidad limitado. El firmware puede ver tu tráfico, pero con HTTPS + VPN la exposición es mínima. Si tu modelo de amenaza incluye estados nacionales o atacantes sofisticados, ahí sí importa más.

Teóricamente sí, el firmware puede:

  • Enviar datos a través del chip WiFi

Lo que necesita para espiarte de verdad:

  1. Que el firmware tenga código malicioso (poco probable pero no verificable)
  2. Que tenga capacidad de exfiltrar datos fuera del chip

Lo que NO puede hacer:

  • Acceder a tu disco
  • Escapar más allá de la interfaz WiFi

El problema con Trisquel

Trisquel bloquea estos firmwares binarios porque:

  • La FSF considera que el firmware no libre = software no libre
  • No podes verificar qué hace realmente ese binario

Mitigaciones prácticas (en Arch)

  1. Usar HTTPS siempre - cifra el tráfico punto a punto, el firmware solo ve datos cifrados
  2. VPN - el firmware ve conexión a 1 IP (tu VPN), no sabe qué navegas
  3. Ethernet si es posible - evita el problema del todo
  4. Tarjetas con firmware más abierto - algunas Intel tienen firmware parcialmente libre

Riesgo real: Bajo para amenazas comunes, mayor si tu modelo incluye atacantes con recursos para合作的 firmware personalizado de Realtek.

Usuario: Y no se puede hacer ingeniería inversa al firmware propietario y darse cuenta si espian o no?

Respuesta: Sí, se puede hacer ingeniería inversa al firmware. De hecho, ya se ha hecho en cierta medida.

Estado actual del RTL8723BE

La comunidad ha analizado este firmware y no se ha encontrado evidencia de espionaje activo. El firmware es relativamente simple y solo hace lo que necesita:

  • Gestionar conexiones WiFi
  • Enviar/recibir paquetes
  • Control de potencia

No tiene capacidades de:

  • Conectarse a servidores remotos por iniciativa propia
  • Almacenar y exfiltrar datos
  • Ejecutar código externo

Por qué es difícil detectar backdoors

  1. El firmware es pequeño (~50-100KB) - tiene poco código para esconder cosas
  2. Se ha auditado parcialmente - la comunidad de reverse engineering ha mirado firmwares Realtek
  3. Un backdoor sería obvio - tendría que conectarse a una IP, hacer requests DNS, etc. - eso deja rastros

Alternativas para mayor seguridad

Si queres máxima paranoia:

  1. Tarjetas Intel WiFi - algunos modelos tienen firmware que se puede modificar/open sourcer parcialmente
  2. USB WiFi con chipset más documentado - algunos tienen más ojos sobre ellos
  3. Usar solo Ethernet - elimina todo el problema

Veredicto para tu caso

El RTL8723BE es un chip muy común, widely used, y no hay reportes creíbles de backdoors documentados. El riesgo real de espionaje por el firmware es muy bajo comparado con:

  • El sistema operativo en sí
  • Las apps que usas
  • El tráfico que envías (si no usas HTTPS)

Si tu modelo de amenaza incluye gobierno/NSA/etc., la preocupación debería estar más en el hardware, el router, o las capas de software, no tanto en el firmware WiFi de una laptop de consumo.

  • AUR - si, hubo casos de malware camuflado en paquetes de AUR ( dependency confusion attacks, etc.)
