Within /etc/portage/ users can create the following files:

  • package.mask which lists the packages that Portage should never try to install
  • package.unmask which lists the packages Portage should be able to install even though the Gentoo developers highly discourage users from emerging them
  • package.accept_keywords which lists the packages Portage should be able to install even though the package hasn’t been found suitable for the system or architecture (yet)
  • package.use which lists the USE flags to use for certain packages without having the entire system use those USE flags

Binarios

Si instalas el binario del grub en Gentoo no es lo mismo que el binario en ArchLinux.
El binario de GRUB no viene con la bandera mount (USE Flag) habilitada por lo tanto si quiero usar OS-PROBER estoy obligado a compilar el GRUB porque el binario de GRUB en Gentoo no tiene la bandera mount activada entonces tengo que compilarlo para poder usar OS-PROBER.
Es decir los binarios vienen con la mínima cantidad de banderas posibles, a diferencia de ArchLinux que vienen con la máxima cantidad de banderas posibles activadas.

Solucion

Los archivos de Gentoo están en /boot de la partición EFI (que es compartida), no en /dev/sda3/boot.

Aquí está la entrada correcta para Gentoo:

sudo vim /etc/grub.d/40_custom

Agrega esto (si cambiás la particion tenes que cambiar el UUID).

menuentry 'Gentoo Linux' --class gentoo --class gnu-linux --class gnu --class os {
    insmod part_gpt
    insmod ext2
    set root='hd0,gpt3'
    if [ x$feature_platform_search_hint = xy ]; then
      search --no-floppy --fs-uuid --set=root --hint-bios=hd0,gpt3 --hint-efi=hd0,gpt3 --hint-baremetal=ahci0,gpt3  78b556fb-5a81-40ab-b8f2-29d5527e3120
    else
      search --no-floppy --fs-uuid --set=root 78b556fb-5a81-40ab-b8f2-29d5527e3120
    fi
    linux (hd0,gpt1)/vmlinuz-6.18.2-p1-gentoo-dist root=UUID=78b556fb-5a81-40ab-b8f2-29d5527e3120 rw
    initrd (hd0,gpt1)/initramfs-6.18.2-p1-gentoo-dist.img
}

La clave es que el kernel y el initramfs están en (hd0,gpt1) que es tu partición EFI /dev/sda1, pero el root filesystem es /dev/sda3.

Por qué falló os-prober:

  1. Kernel en partición EFI compartida: os-prober detectó que Gentoo está en /dev/sda3, pero no se dio cuenta de que el kernel de Gentoo está en /dev/sda1 (la partición EFI compartida). Asumió que el kernel estaría en /boot dentro de /dev/sda3.

  • Los nombres de kernel no siguen convenciones estándar

Since you mentioned using systemctl, you are likely using the systemd init system. The reason your current enable commands didn’t work is that wpa_supplicant and dhcpcd need to be tied to your specific network interface (wlp1s0).

Here is how to fix this for both init systems (Systemd and OpenRC), as Gentoo users often switch between them.

Option A: If you are using Systemd

You need to enable the service instances specifically for your wireless interface.

  1. Enable wpa_supplicant for your interface: Instead of just wpa_supplicant, enable the template service that includes your interface name.

    sudo systemctl enable wpa_supplicant@wlp1s0.service

  2. Enable dhcpcd: The standard dhcpcd.service usually works, but to be specific to your interface (and avoid it trying to manage other interfaces like ethernet if you don’t want it to), you can often rely on the global service. If you previously enabled the generic one, disable it first to be clean, then ensure it’s running:

    sudo systemctl enable dhcpcd.service

    (Note: If dhcpcd still fails to pick up the interface, it is because wpa_supplicant hasn’t associated with the Wi-Fi yet. Systemd handles these dependencies automatically if configured correctly, but sometimes it is cleaner to let the networkmanager handle both, or stick to the manual binding below.)

A common Systemd alternative (Cleaner): If the above gives you trouble, the most reliable “manual” method on systemd is to bind them together.

  1. Run sudo systemctl disable dhcpcd.service (to stop the global one).
  2. Enable the interface-specific dhcpcd: sudo systemctl enable dhcpcd@wlp1s0.service.
  3. Ensure wpa_supplicant is enabled as shown in step 1 above.

Reboot to test.

Option B: If you are using OpenRC (Gentoo Default)

If you are actually using OpenRC (which uses rc-update, not systemctl) but installed systemd tools by mistake, here is the correct OpenRC way to do this. This is the “Gentoo Way” of handling networks.

  1. Create a network script symlink: Gentoo uses a generic net.lo script. You need to link it to your wireless interface.

    sudo ln -s /etc/init.d/net.lo /etc/init.d/net.wlp1s0

  2. Configure /etc/conf.d/net: Tell the system to use wpa_supplicant for this interface. Add these lines to the file:

    # /etc/conf.d/net
modules="wpa_supplicant"
wpa_supplicant_wlp1s0="-Dnl80211 -c/etc/wpa_supplicant/wpa_supplicant.conf"
config_wlp1s0="dhcp"

  3. Enable the scripts to start at boot:

    sudo rc-update add net.wlp1s0 default

    (Note: You usually do not need to add dhcpcd or wpa_supplicant separately to the runlevel when using this method; the net.wlp1s0 script handles calling them).

  4. Start it immediately (optional):

    sudo /etc/init.d/net.wlp1s0 start

Summary checklist

  • Systemd: Use wpa_supplicant@wlp1s0.service.
  • OpenRC: Use the net.wlp1s0 symbolic link method.

Important Note: Do not mix the two methods. If you use the OpenRC net.wlp1s0 method, do not enable the standalone wpa_supplicant or dhcpcd services. If you use Systemd, use the service files directly.
